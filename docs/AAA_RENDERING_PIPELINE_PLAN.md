# SelfEngine AAA Rendering Pipeline Plan

## Goal

Bring SelfEngine from the current lean Forward 3D renderer toward a mainstream PC/console AAA rendering stack that can preview real Unreal Engine project art, not just self-contained demo scenes. The long-term product goal is to point SelfEngine at `D:\UEProject`, open real UE5 project folders and art-content sets such as `D:\UEProject\乌云庙宇 5.8`, enumerate their maps/scenes/assets, and render them with visual parity close enough to the Unreal Engine 5 editor viewport that artists can use SelfEngine as a trustworthy preview target.

The renderer work therefore serves an asset-compatibility and visual-parity target: HDR deferred rendering with a forward residual/Forward+ path, tiled/clustered lighting, physically based materials, image-based lighting, reflection probes, production shadow atlases, screen-space AO/reflections, probe lighting, volumetrics, temporal anti-aliasing/upscaling, scalable post processing, order-independent translucency, and production-grade diagnostics. Benchmark grids and hand-authored demos are regression tools only; they are not the finish line.

The final product must behave like a practical UE5-content previewer: choose a real project/content root, inspect its maps and asset dependencies, load a scene through a reproducible bridge/import path, and make the SelfEngine viewport comparable to the UE5 editor viewport from the same camera. A renderer-only showcase that cannot open `D:\UEProject` content, cannot identify the source UE assets it is displaying, or cannot explain visual mismatches against UE5 is considered off-goal even if the image looks impressive.

This plan intentionally prioritizes the mature rendering systems used broadly across shipping high-end games before tackling extreme frontier systems. Nanite-class virtualized geometry, Lumen-class fully dynamic GI, full Virtual Shadow Maps, hardware ray tracing, and a native TSR-class upscaler are useful research targets, but they are explicitly out of scope for the current implementation track unless a target `D:\UEProject` asset requires a fallback strategy for them. Current SelfEngine strengths are a clean Vulkan base, PBR-ish forward shader, model import, single shadow map, culling/instancing, ImGui controls, and deterministic benchmarks. The immediate weakness is end-to-end production content carrying capacity: project discovery, asset conversion, scene reconstruction, passes, attachments, descriptors, timings, debug views, and visual-diff evidence are still tied to one legacy command-buffer/demo path.

Mainstream AAA anchors:

- Hybrid HDR renderer: deferred opaque lighting plus forward residual/Forward+ for translucency, particles, water, hair, special materials, and debug overlays.
- Lighting core: directional, point, spot, rect, emissive hints, tiled/clustered culling, BRDF LUT, sky irradiance, reflection probes, and local reflection fallback.
- Shadowing: cascaded directional shadows, spot/point shadow atlases, PCF/PCSS/EVSM quality tiers, contact shadows, stable filtering, and low/medium/high quality presets.
- Indirect/reflection tier: SSAO/GTAO, SSR, baked/probe GI, irradiance/light probes, reflection probes, temporal accumulation, denoisers, and clear fallback behavior.
- Atmosphere/post/transparency: volumetric fog, sky/atmosphere, bloom, tone mapping, color grading, DOF, motion blur, TAA/TAAU or FSR-style upscaling, weighted blended translucency, and eventually OIT.
- Unreal project preview parity: discover `.uproject` roots under `D:\UEProject`, ingest exported or bridge-converted meshes/materials/textures/maps, reconstruct levels, lights, cameras, sky/post-process volumes, and compare SelfEngine screenshots against UE5 viewport references.
- Production diagnostics: frame graph visibility, debug views, pass/resource timings, benchmark CSVs, RenderDoc markers, target-project scene captures, visual diffs, and measurable quality presets.
- Third-party leverage: use mature, well-maintained libraries and SDKs where they materially reduce implementation time or risk. SelfEngine should not hand-roll complex importers, compression codecs, denoisers, upscalers, shader tools, or visual-diff infrastructure when a proven library can be integrated cleanly behind an engine-owned boundary.

## Product Acceptance Target

The AAA renderer is considered successful only when it can load and preview real target project content, not when it renders an isolated benchmark demo.

- Target content root: `D:\UEProject`, with `D:\UEProject\乌云庙宇 5.8` as the first named art-content benchmark. If the on-disk folder name changes, update this plan and the bridge manifest instead of replacing it with an internal demo.
- Minimum project-preview flow: select a project/content root, enumerate maps/scenes/assets, open a chosen scene or exported level, stream/load referenced meshes/materials/textures, reconstruct transforms/lights/cameras/sky/post-process state, and render through the same mainstream AAA pipeline used by engine scenes.
- Visual parity target: for selected reference cameras, produce SelfEngine screenshots and UE5 editor viewport screenshots with documented differences in exposure, tone mapping, material response, shadowing, reflections, fog, translucency, anti-aliasing, and post effects. Each milestone should shrink the gap or record a specific unsupported UE feature/fallback.
- UE viewport parity gate: a milestone is not accepted as complete until it either renders at least one target-project scene/view with a UE5 reference image, or records why that scene is currently blocked by a named bridge/import/rendering limitation.
- Asset bridge policy: direct `.uasset/.umap` support is not assumed as a first step. A practical bridge may use UE editor automation, Datasmith/glTF/USD/FBX export, asset registry metadata, or a generated manifest, but the SelfEngine runtime must treat the result as project content with stable asset ids rather than a handcrafted demo scene.
- Third-party policy: importing a capable library is allowed when it accelerates real UE-content preview, keeps licensing acceptable for the project, and is wrapped behind a small SelfEngine interface. Prefer established tools for formats and rendering support work, such as Assimp/tinygltf or equivalent glTF loaders, OpenUSD where useful, meshoptimizer, KTX-Software/Basis Universal, DirectXTex-style texture tooling when practical, SPIRV-Reflect/SPIRV-Cross, RenderDoc markers, FSR-style upscaling SDKs, ImGuizmo/editor widgets, and image-diff libraries. Do not block progress by reimplementing these systems prematurely.
- Demo policy: `SelfEngineForward3D` grid scenes, black-hole scenes, and synthetic benchmark materials remain smoke/regression tests. They cannot be used as the final proof that the renderer satisfies the AAA plan.

## Target Frame Pipeline

The final renderer should converge on this pass family:

1. Project/content acquisition: discover `D:\UEProject` projects, read bridge manifests, import/exported meshes/materials/textures/maps, preserve stable asset ids, and report unsupported UE asset classes.
2. Scene reconstruction: load levels, transforms, component hierarchy, cameras, lights, reflection captures, sky/fog/post-process volumes, collision-independent bounds, and per-map render settings.
3. Frame setup and scene update: camera history, jitter, exposure state, light lists, render settings, GPU budget.
4. Visibility and geometry preparation: CPU frustum culling, GPU hierarchical Z, occlusion culling, instance compaction, classic LOD selection, material bucketing.
5. Depth and velocity: depth prepass where useful, motion vectors for all moving camera/object paths, previous-frame transforms.
6. Geometry scalability path: batched classic meshes, instancing, meshlet-ready asset metadata where cheap, CPU/GPU culling, LOD transitions, optional visibility-buffer experiments after the deferred path is stable.
7. Shadowing: cascaded directional shadow maps, spot/point shadow atlases, PCF/PCSS/EVSM/contact-shadow quality tiers, stable filtering, cache/fallback policy.
8. GBuffer: albedo, normals, roughness/metallic/AO, emissive, velocity, material id, depth; opaque PBR defaults to deferred.
9. Lighting: tiled/clustered light culling, direct PBR lighting, image-based lighting, sky/atmosphere, shadow resolve.
10. Indirect lighting/reflections: SSAO/GTAO, SSR, reflection probes, irradiance/light probes, optional baked probe data, temporal denoise, explicit fallback tiers.
11. Forward+ path: weighted blended translucency/OIT carrier, particles, hair/fur, special materials, black-hole effect, water, debug overlays.
12. Volumetrics and atmosphere: fog, clouds, light shafts, volumetric shadows, aerial perspective.
13. Post processing: HDR scene color, auto exposure, bloom, tone mapping, color grading, DOF, motion blur, lens effects.
14. Temporal AA/upscaling: TAA first, TAAU/FSR-style upscaler second, dynamic resolution, sharpening, history rejection.
15. UI/debug/present: project browser, scene viewport, ImGui, visualizers, capture markers, screenshot/benchmark output, UE-reference visual comparison.

## Execution Roadmap

### Phase A: Unreal Project Asset Bridge And Visual Baseline

- Add a project-browser path for `D:\UEProject` that can enumerate project/content roots, detect `.uproject` files or bridge manifests, and list candidate maps/scenes/assets.
- Define a bridge manifest format that records source UE project path, exported mesh/material/texture paths, map hierarchy, actor transforms, lights, cameras, sky/fog/post-process settings, and stable source asset ids.
- Add an editor/export automation target outside the runtime when direct `.uasset/.umap` parsing is not practical. Acceptable first bridge outputs are glTF/USD/FBX plus sidecar JSON for UE-specific metadata.
- Prefer existing export/import libraries and UE automation APIs for bridge work. The engine-owned value is stable asset identity, diagnostics, scene reconstruction, and visual parity; low-level format decoding should use maintained libraries where possible.
- Create the first target-project smoke for `D:\UEProject\乌云庙宇 5.8`: discover the content root, produce or consume a bridge manifest, load at least one representative scene, and record missing asset/material feature counts.
- Add a UE-reference baseline workflow for that target: store matching UE5 editor viewport camera data, exposure/post-process settings, screenshots, and SelfEngine captures so the comparison is against the real content, not against a recreated approximation.
- Add reference capture slots: store UE5 viewport screenshots and camera metadata beside SelfEngine screenshots so every renderer milestone can be visually compared against real content.
- Add asset-compatibility diagnostics: unsupported material nodes, missing textures, fallback shaders, unsupported lights, unsupported post-process settings, import warnings, and source asset ids must be visible in UI and CSV/log output.

### Phase 0: Render-Graph Carrying Capacity

- Add an explicit frame graph/pass description layer with names, queues, resources, active/roadmap status, and debug visibility.
- Keep the existing Forward 3D output unchanged while describing current Shadow/Main/Overlay/ImGui/Present passes through the new layer.
- Extend stats and ImGui so every future feature lands as an observable pass before becoming a complex Vulkan implementation.
- Keep legacy render pass objects alive for now; introduce dynamic rendering/transient attachments only after the graph vocabulary is in place.

### Phase 1: HDR Deferred Foundation

- Introduce render targets for HDR scene color, depth, velocity, and a compact GBuffer.
- Treat imported target-project meshes/materials as the reference workload for deferred parity; synthetic grid materials are only regression coverage.
- Start with a resource-plan contract inside the frame graph: legacy physical resources and future HDR/GBuffer/history/cache resources must be visible in ImGui and benchmark CSV before Vulkan attachment migration starts.
- Allocate a real per-swapchain-image HDR scene color target first, then migrate the rendering path to write/resolve through it in a later slice.
- Add an HDR offscreen render pass and framebuffer, then record a clear-only HDR pass before scene rendering migrates, so pipeline/framebuffer lifetime can be validated separately from visual output changes.
- Allocate the first deferred target set before the GBuffer pass exists: scene depth, velocity, albedo, normal/roughness, and material buffers.
- Add a clear-only GBuffer render pass and framebuffer over those targets before migrating opaque geometry.
- Add a first GBuffer 3D geometry pipeline that writes opaque albedo, normal/roughness, material parameters, velocity placeholder, and depth while legacy forward still owns visible output.
- Add a first fullscreen deferred-lighting pass that samples the GBuffer and writes HDRSceneColor; keep swapchain presentation on legacy forward until debug/composite parity is proven.
- Add an HDR composite/debug path that samples HDRSceneColor, tone maps it, and can replace the legacy forward scene pass through a debug view or benchmark environment switch.
- Add GBuffer debug visualizers for albedo, normal, roughness, metallic, depth, velocity, and deferred shadow visibility so deferred correctness can be inspected before it becomes the default path.
- Add the first deferred shadow parity slice: reconstruct world position from depth using inverse frame matrices, sample the existing directional shadow map from deferred lighting, and keep the legacy single shadow map as the fallback tier until the production shadow-atlas work starts.
- Split opaque 3D rendering into GBuffer generation plus deferred lighting, while preserving Forward 3D as a fallback/debug path.
- Move material data from push-constant-heavy layout toward GPU material buffers with stable material ids.
- Add debug views for GBuffer channels, velocity, depth, light complexity, and overdraw.

### Phase 2: Modern Lighting Core

- Add scene light components for directional, point, spot, rect, and emissive-light hints.
- Map bridge-imported UE light actors and sky/atmosphere parameters into scene-owned light components with explicit unsupported-field diagnostics.
- Add the first scene-owned primary directional light to `Scene3D`, with renderer collection into `FrameLightSet` and material-push-constant fallback kept only for compatibility paths.
- Add the first scene-owned point light data model to `Scene3D`, with light revision tracking and renderer collection into `FrameLightSet`.
- Add the first scene-owned spot light data model to `Scene3D`, encoded through local-light slots with direction/type and cone-angle payloads.
- Add the first scene-owned rect light data model to `Scene3D`, encoded through local-light slots with direction/type and width/height payloads as a first-cut area-light approximation.
- Move the first directional-light data out of material push constants and into renderer-owned frame light constants so forward, deferred, shadow projection, and debug views read one authoritative light direction/intensity source.
- Introduce an explicit renderer light scaffold (`RendererLightKind`, `RendererDirectionalLight`, and `FrameLightSet`) before GPU light buffers. This gives Phase 2 a real ownership boundary and exposes total/directional/local light counts in diagnostics.
- Upload the `FrameLightSet` into a renderer-owned GPU-visible storage buffer bound from the frame descriptor set. The first buffer tier carries one directional light, ambient/specular controls, frame light counts, and fixed local-light records used by deferred lighting.
- Add a first CPU-populated tiled light assignment scaffold over the local-light record array, then replace the population step with compute culling and feed both deferred and Forward+ passes.
- Replace the single hard-coded directional light material path with renderer-owned light buffers.
- Add environment lighting: sky irradiance, reflection probes, prefiltered cubemaps, BRDF LUT, and local reflection fallback.

### Phase 3: Production Shadowing

- Upgrade the single directional shadow map into cascaded shadow maps with stable splits, per-cascade texel snapping, bias controls, and debug views.
- Use target-project sun/sky and local-light scenes as the quality reference for shadow softness, contact grounding, cascade stability, and atlas budgets.
- Add spot and point shadow atlas support with per-light resolution budgets, cube/dual-paraboloid policy decided by quality tier, and atlas occupancy diagnostics.
- Add filtering tiers: baseline PCF, higher-quality PCSS-style soft shadows, EVSM/variance-shadow-map experiments where format support is acceptable, and contact shadows for near-field grounding.
- Preserve current single shadow map as the Low/Fallback tier until cascades and local-light shadow atlases are benchmark-proven.

### Phase 4: Screen-Space Reflections/AO + Probe Lighting

- Add SSAO first, then deepen it toward GTAO with normal/depth-aware filtering, temporal stability, and debug views.
- Add SSR for glossy reflections, with roughness-aware tracing, thickness controls, hierarchical depth traversal, temporal accumulation, and reflection-probe fallback.
- Add sky irradiance, BRDF LUT, prefiltered environment maps, local reflection probes, and probe blending volumes.
- Import or approximate UE reflection captures, skylight cubemaps, lightmass/probe metadata, and post-process reflection settings from the bridge manifest.
- Add a probe GI/light-probe tier for static or slowly changing indirect diffuse lighting before considering fully dynamic GI research.

### Phase 5: Mainstream Geometry/Visibility Scalability

- Convert imported meshes into an engine-native cache with stable mesh/material ids, LOD groups, bounds, material ranges, and streaming-friendly chunks.
- Preserve UE source asset ids, LOD screen-size thresholds where available, lightmap/UV-channel metadata, collision-independent render bounds, and instance grouping from the bridge manifest.
- Add classic LOD selection, hysteresis/fading policy, instancing, GPU-driven indirect draw preparation where it reduces CPU cost, and hierarchical occlusion using depth/Hi-Z.
- Add meshlet metadata only as a compatibility-ready asset format, not as a Nanite-style virtualized renderer.
- Keep classic mesh rendering for skeletal meshes, translucent meshes, debug geometry, and unsupported materials.

### Phase 6: Temporal AA/Upscaling And Post

- Add velocity-correct TAA, then dynamic-resolution temporal upscaling.
- Add a TAAU/FSR-style plugin boundary with history, reactive masks where useful, exposure, jitter control, sharpening, and quality presets.
- Move tone mapping, exposure, bloom, color grading, motion blur, DOF, and sharpening into graph passes.
- Match target-project UE viewport exposure, tone-mapper, color grading, fog, bloom, DOF, and motion-blur settings closely enough that screenshot deltas are attributable to known unsupported features rather than arbitrary SelfEngine defaults.

### Phase 7: Production Project Preview Tooling

- Add RenderDoc-friendly debug names and capture markers for every pass/resource.
- Add a real project viewport mode: project tree, map list, source asset inspector, missing-feature report, camera bookmarks, UE-reference screenshot selection, and SelfEngine screenshot capture.
- Add benchmark scenes: target-project maps from `D:\UEProject`, city-scale static mesh, many lights, shadow-atlas stress, SSR/AO/probe-lighting stress, black-hole full-screen stress, heavy post/upscale, and translucency/OIT stress.
- Add CSV columns per pass, per resource pool, GPU memory, shadow atlas occupancy, LOD/instance counts, light count, probe count, history resets, dynamic-resolution scale, source asset count, missing asset count, fallback material count, unsupported UE feature count, and visual-diff summary metrics.
- Add quality presets: Cinematic, Epic, High, Medium, Low, and FPS-first. Every expensive system must have a measurable fallback.

### Advanced Research Backlog

These systems are intentionally postponed until the mainstream AAA track above is usable, benchmarked, and debuggable:

- Nanite-class virtualized geometry: cluster hierarchy, cluster streaming, software/compute visibility, visibility buffers, material resolve, and automatic LOD at film-scale triangle counts.
- Lumen-class dynamic GI/reflections: surface cache/card capture, signed-distance-field or BVH tracing, software/hardware ray tracing tiers, temporal denoise, and fully dynamic multi-bounce lighting.
- Full Virtual Shadow Maps: virtual page tables, physical page atlases, page marking, invalidation/caching, directional clipmaps, local-light pages, and Nanite-scale scene integration.
- Hardware ray tracing path: BLAS/TLAS build policy, shader binding table management, ray-traced shadows/reflections/GI, denoisers, and raster fallbacks.
- Native TSR-class upscaler: deep history rejection, reactive/transparency masks, exposure-coupled resolve, async-compute scheduling, and dynamic-resolution integration beyond the first TAAU/FSR-style path.

## Implementation Rules

- Keep current demos running at every phase: `SelfEngine`, `SelfEngineForward3D`, and `SelfEngineBlackHole`.
- Keep target-project preview working as soon as Phase A exists; after that point, every major rendering phase must run on at least one `D:\UEProject` scene, not only on synthetic demos.
- Treat the UE5 project viewport as the product reference. New lighting, material, shadow, post, translucency, visibility, and temporal features must include a path to validate against imported `D:\UEProject` content, with screenshots/CSV/log diagnostics tied back to the source UE map, actor, component, material, or texture.
- Do not close a phase with only a pretty SelfEngine-controlled scene. A phase closes only when the target-project scene either improves toward UE5 viewport parity or produces a concrete blocker list for the exact UE features that remain unsupported.
- Do not remove the existing Forward 3D path until the deferred path can render imported textured models, shadows, debug views, and benchmark scenes.
- Do not treat a handcrafted benchmark scene as proof of final AAA success. Benchmarks prove regressions and isolated systems; target-project scene preview plus UE-reference visual comparison proves product progress.
- Preserve source asset identity through import, renderer material records, diagnostics, screenshots, and CSV/log output so a visual mismatch can be traced back to the originating UE asset/material/light/map.
- Any unsupported UE material node, light type, post-process field, texture format, geometry feature, or scene component must produce an explicit fallback record instead of silently becoming a generic material or disappearing.
- Prefer adding thin renderer-owned concepts first: pass descriptions, render targets, material ids, light ids, history ids. Add heavyweight GPU algorithms only after the interfaces exist.
- Every phase must add a visible debug mode or benchmark metric; no invisible renderer rewrite counts as complete.
- Use Vulkan 1.3 as the baseline and gate optional mainstream paths behind feature checks: descriptor indexing, dynamic rendering, synchronization2, subgroup operations, and async compute. Mesh shaders and ray tracing remain advanced-research feature gates, not current-track requirements.
- Third-party libraries are acceptable by default when they are mature, reasonably licensed, actively maintained, and hidden behind narrow SelfEngine-owned interfaces. Add them through CMake/FetchContent or local third-party folders in a way that keeps builds reproducible, and record the reason for adoption in this plan or a small ADR when the dependency affects a major subsystem.
