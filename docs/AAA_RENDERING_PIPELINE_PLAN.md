# SelfEngine AAA Rendering Pipeline Plan

## Latest Completed Slice

AMD FidelityFX SSSR has advanced through the official DNSR ResolveTemporal
bridge and into the first controlled visible Deferred composite. The engine
vendors the SSSR/DNSR/SPD shader subset, compiles eight FFX HLSL compute
shaders to SPIR-V, creates a scene-independent `FidelityFxSssrAdapter`, and
dispatches `ClassifyTiles.hlsl`, `PrepareIndirectArgs.hlsl`,
`PrepareBlueNoiseTexture.hlsl`, `Intersect.hlsl`, `Reproject.hlsl`,
`Prefilter.hlsl`, and `ResolveTemporal.hlsl` when
`SE_SSR_BACKEND=ffx-sssr`.

The bridge allocates the vendor-shaped constants buffer plus per-swapchain ray
counter, ray list, denoiser tile list, extracted roughness, variance placeholder,
intersection output, generated blue-noise texture, indirect-args resources,
real history images, Reproject output images for reprojected radiance,
average radiance, variance, and sample count, plus Prefilter output images for
prefiltered radiance, variance, and sample count. ResolveTemporal now consumes
extracted roughness, average radiance, prefiltered radiance/variance/sample
count, reprojected radiance, and the denoiser tile list, then writes the
RadianceHistory, VarianceHistory, and SampleCountHistory images consumed by the
next Reproject pass. AMD typed buffers are bound as
`VK_DESCRIPTOR_TYPE_STORAGE_TEXEL_BUFFER` or
`VK_DESCRIPTOR_TYPE_UNIFORM_TEXEL_BUFFER` with `VK_FORMAT_R32_UINT`. The
`RWTexture2D<float4>` intersection output remains
`VK_FORMAT_R32G32B32A32_SFLOAT`; Reproject's RGB radiance outputs are stored in
RGBA32F images with a documented Vulkan compatibility shader patch that writes
`.xyzz`, because RGB32F sampled/storage images are not portable on the current
device. `PrepareBlueNoiseTexture` uses the official AMD 128x128 1spp
Sobol/ranking/scrambling tables and writes a `VK_FORMAT_R32G32_SFLOAT`
blue-noise texture. `Intersect` runs after HDR lighting so it can consume the
lit scene color mip0 view, depth pyramid, GBuffer normal, extracted roughness,
environment map, generated blue noise, ray list, and prepared indirect ray
count. Reproject then consumes the denoiser tile list and the second indirect
dispatch record at byte offset `12`; Prefilter and ResolveTemporal use the same
denoiser indirect record. ResolveTemporal writes the current swapchain history
slot and the bridge copies that FFX history state to the other slots so the next
frame's Reproject and Deferred reflection composite have a coherent starting
point. The Deferred consumer now binds the previous completed FFX
`RadianceHistory` as binding 17 when `SE_SSR_BACKEND=ffx-sssr`, a valid
previous temporal frame exists, and the FFX history has completed at least once.
The shader marks this with the `131072` SSR control bit so it treats the source
as radiance history instead of internal `SSRResolved` alpha-confidence metadata,
then blends it conservatively over the probe/IBL fallback.

Validation: `scripts\Test-FidelityFxSssrIntegration.ps1 -SkipBuild -Strict
-OutputDirectory tmp\ffx_sssr_deferred_composite_bridge` passed `125 pass / 0
fail`, with LightingShowcase and Forward3D FBX FFX lanes reporting backend
requested/active `1/1`, runtime active `1`, deferred composite
requested/active/descriptor/history/source `1/1/1/1/1`, `radianceSource=4`,
FrameGraph validation `0`, and Vulkan validation diagnostics `0`. The
internal-backend control reports requested/active `0/0`, deferred composite
`0/0/0/0/0`, `radianceSource=2`, FrameGraph validation `0`, and Vulkan
validation diagnostics `0`. Existing SSR regression
`scripts\Test-SsrRefinementHealth.ps1 -SkipBuild -SkipSigning -Strict
-OutputDirectory tmp\ssr_regression_after_ffx_deferred_composite` passed
`691 pass / 0 fail`.

The next SSR slice is visual acceptance of the controlled FFX composite and then
quality refinement of the visible SSSR contribution, keeping the stable
probe/IBL fallback as the off-screen and low-confidence baseline.

## Goal

Bring SelfEngine from the current lean Forward 3D renderer toward a complete mainstream PC/console AAA rendering stack first. The current execution priority is renderer capability completion. Do not schedule new UE5 scene parsing, Unreal automation, project browser, bridge-manifest expansion, or UE viewport-parity work while the renderer gaps below remain open.

Current directive: treat SelfEngine as a standalone renderer first. The next milestones must fill missing mainstream 3A rendering capabilities, not chase `D:\UEProject`, `.uasset/.umap` parsing, UE scene reconstruction, or UE viewport matching. UE-derived assets may only be used as optional imported-model stress content if they have already been exported into ordinary formats and do not require new bridge work.

The renderer work now serves a renderer-completeness target: HDR deferred rendering with a forward residual/Forward+ path, tiled/clustered lighting, physically based materials, image-based lighting, reflection probes, production shadow atlases, screen-space AO/reflections, probe lighting, volumetrics, temporal anti-aliasing/upscaling, scalable post processing, order-independent translucency, visibility/LOD scalability, and production-grade diagnostics. Benchmark grids, imported standalone models, and hand-authored stress scenes are valid first-class validation targets while the renderer itself is being completed.

UE5 project/content preview remains a later product-integration track. Existing bridge work can stay as frozen regression coverage, but current milestones must not be blocked on parsing `.uasset/.umap`, reconstructing UE levels, exporting Unreal scenes, or matching the UE editor viewport. Once the mainstream AAA renderer is feature-complete enough, the UE project bridge can return as a validation and content-ingestion layer on top of that renderer.

This plan intentionally prioritizes the mature rendering systems used broadly across shipping high-end games before tackling extreme frontier systems. Nanite-class virtualized geometry, Lumen-class fully dynamic GI, full Virtual Shadow Maps, hardware ray tracing, and a native TSR-class upscaler are useful research targets, but they are explicitly out of scope for the current implementation track. Current SelfEngine strengths are a clean Vulkan base, PBR-ish forward shader, model import, single shadow map, culling/instancing, ImGui controls, deterministic benchmarks, and the first slices of deferred lighting/shadow/probe work. The immediate weakness is renderer feature completeness and polish: passes, attachments, descriptors, material payloads, lighting quality, temporal stability, post effects, visibility scaling, timings, debug views, and visual evidence still need to converge into one production-grade path.

Mainstream AAA anchors:

- Hybrid HDR renderer: deferred opaque lighting plus forward residual/Forward+ for translucency, particles, water, hair, special materials, and debug overlays.
- Lighting core: directional, point, spot, rect, emissive hints, tiled/clustered culling, BRDF LUT, sky irradiance, reflection probes, and local reflection fallback.
- Shadowing: cascaded directional shadows, spot/point shadow atlases, PCF/PCSS/EVSM quality tiers, contact shadows, stable filtering, and low/medium/high quality presets.
- Indirect/reflection tier: SSAO/GTAO, SSR, baked/probe GI, irradiance/light probes, reflection probes, temporal accumulation, denoisers, and clear fallback behavior.
- Atmosphere/post/transparency: volumetric fog, sky/atmosphere, bloom, tone mapping, color grading, DOF, motion blur, TAA/TAAU or FSR-style upscaling, weighted blended translucency, and eventually OIT.
- Renderer-first validation: stress the same pipeline with built-in benchmark scenes, imported standalone models, high-light-count scenes, material stress scenes, shadow/probe/SSR/AO cases, volumetric/post-process cases, and captured visual baselines.
- Production diagnostics: frame graph visibility, debug views, pass/resource timings, benchmark CSVs, RenderDoc markers, renderer reference captures, visual diffs, and measurable quality presets.
- Third-party leverage: use mature, well-maintained libraries and SDKs where they materially reduce implementation time or risk. SelfEngine should not hand-roll complex importers, compression codecs, denoisers, upscalers, shader tools, or visual-diff infrastructure when a proven library can be integrated cleanly behind an engine-owned boundary.

## Renderer Acceptance Target

The current AAA renderer milestone is considered successful when SelfEngine has the mainstream 3A rendering capabilities expected from a modern raster renderer, with measurable debug and benchmark coverage. UE5 scene parsing, project preview, Unreal export automation, and UE viewport matching are explicitly not part of the current acceptance gate.

- Main visible path: deferred opaque HDR lighting plus forward residual/Forward+ for transparent, special, particle-like, water-like, and debug materials.
- Materials: stable material ids, PBR texture payloads, normal/tangent parity, emissive, alpha mask/blend, clearcoat, transmission/volume placeholders, texture transforms, double-sided policy, and fallback diagnostics.
- Lighting and IBL: directional, point, spot, rect approximations or LTC, tiled/clustered light culling, BRDF LUT, sky irradiance, reflection probes, local probe blending, and explicit global fallback behavior.
- Shadows: CSM with stable splits/blend/fade/filtering, local point/spot shadow atlases, contact shadows, cache/invalidation policy, debug views, and quality presets.
- Indirect/reflection: SSAO moving toward GTAO, SSR with color/history/denoise tiers, probe fallback and blending, and a static probe/light-probe tier before any fully dynamic GI research.
- Atmosphere/post/temporal: volumetric fog/atmosphere, bloom, tone mapping, exposure, color grading, DOF, motion blur, TAA, TAAU/FSR-style upscaling, dynamic resolution, and sharpening.
- Visibility/scalability: classic LODs, instancing, CPU/GPU culling, Hi-Z/occlusion scaffolding, instance compaction, material bucketing, and GPU/CPU cost metrics.
- Diagnostics: every expensive feature must expose ImGui controls, render-debug views, frame-graph passes/resources, CSV metrics, quality preset state, and off/fallback behavior.
- Third-party policy: importing a capable library is allowed when it accelerates renderer completion, keeps licensing acceptable for the project, and is wrapped behind a small SelfEngine interface. Prefer established tools for formats and rendering support work, such as Assimp/tinygltf or equivalent loaders, meshoptimizer, KTX-Software/Basis Universal, DirectXTex-style texture tooling when practical, SPIRV-Reflect/SPIRV-Cross, RenderDoc markers, FSR-style upscaling SDKs, ImGuizmo/editor widgets, and image-diff libraries. Do not block progress by reimplementing these systems prematurely.
- UE bridge policy: existing UE bridge/project-preview code may remain, but it is frozen. Only keep it compiling when renderer work touches shared code. Do not add new UE scene parsing, export automation, manifest fields, project-browser behavior, or UE visual-parity gates until the renderer-first acceptance target above is substantially complete.

UE work may re-enter the plan only after this renderer gate is met:

- Deferred HDR is the default visible path for renderer-owned scenes, with forward residual/Forward+ used for transparency, special effects, water-like materials, and debug overlays.
- PBR materials, IBL, reflection probes, shadows, SSAO/GTAO, SSR, volumetrics, post, temporal AA/upscaling, LOD/occlusion, and diagnostics all have on/off controls, fallback behavior, CSV/frame-graph evidence, and smoke coverage.
- Imported generic assets (`.gltf/.glb/.fbx/.obj` or equivalent) render through the same material, lighting, shadow, post, and temporal paths as built-in benchmark scenes.
- Quality presets and reference captures make visual regressions measurable without relying on UE editor screenshots.

## Target Frame Pipeline

The final renderer should converge on this pass family:

1. Frame setup and scene update: camera history, jitter, exposure state, light lists, render settings, quality preset, and GPU budget.
2. Visibility and geometry preparation: CPU frustum culling, GPU hierarchical Z, occlusion culling, instance compaction, classic LOD selection, material bucketing, and draw preparation.
3. Depth and velocity: depth prepass where useful, motion vectors for camera/object motion, previous-frame transforms, and history-reset diagnostics.
4. Geometry scalability path: batched classic meshes, instancing, meshlet-ready metadata where cheap, CPU/GPU culling, LOD transitions, and optional visibility-buffer experiments after the deferred path is stable.
5. Shadowing: cascaded directional shadow maps, spot/point shadow atlases, PCF/PCSS/EVSM/contact-shadow quality tiers, stable filtering, cache/fallback policy.
6. GBuffer: albedo, normals, roughness/metallic/AO, emissive, velocity, material id, depth; opaque PBR defaults to deferred.
7. Lighting: tiled/clustered light culling, direct PBR lighting, image-based lighting, sky/atmosphere, shadow resolve.
8. Indirect lighting/reflections: SSAO/GTAO, SSR, reflection probes, irradiance/light probes, optional baked probe data, temporal denoise, explicit fallback tiers.
9. Forward+ path: weighted blended translucency/OIT carrier, particles, hair/fur, special materials, black-hole effect, water, debug overlays.
10. Volumetrics and atmosphere: fog, clouds, light shafts, volumetric shadows, aerial perspective.
11. Post processing: HDR scene color, auto exposure, bloom, tone mapping, color grading, DOF, motion blur, lens effects.
12. Temporal AA/upscaling: TAA first, TAAU/FSR-style upscaler second, dynamic resolution, sharpening, history rejection.
13. UI/debug/present: scene viewport, ImGui, visualizers, capture markers, screenshot/benchmark output, frame graph, and visual-diff tooling.

## Execution Roadmap

### Active Track: Complete The Mainstream AAA Renderer

Until the renderer acceptance target is met, every new implementation slice should close one of these renderer capability gaps and include visible diagnostics:

- Deferred/HDR default readiness: imported-model material parity, real velocity, robust depth handoff, exposure/tone-map consistency, and fallback-safe promotion from legacy forward.
- Lighting and IBL: BRDF LUT, sky irradiance, prefiltered cubemap path, reflection-capture resources, multi-probe blending, parallax/box projection, and real LTC rect lighting.
- Production shadows: CSM stability/filters, local point/spot atlas quality, cache invalidation, contact-shadow stability, quality presets, and visual-diff evidence.
- Screen-space/probe indirect: SSAO to GTAO, SSR history/denoise, probe fallback hierarchy, static irradiance/light probes, and unsupported-feature fallback reporting.
- Atmosphere, volumetrics, and post: height/distance fog, volumetric fog, sky/atmosphere, bloom, exposure, color grading, DOF, motion blur, lens effects, and graph-level controls.
- Temporal and upscaling: motion vectors, TAA, TAAU/FSR-style plugin boundary, dynamic resolution, sharpening, history rejection, and reset diagnostics.
- Geometry and visibility scalability: engine-native mesh cache, classic LODs, instance batching, GPU/Hi-Z occlusion, material bucketing, indirect draw preparation, and CPU/GPU cost metrics.
- Renderer tooling: benchmark scenes, RenderDoc markers, debug views, CSV/frame-graph coverage, reference captures, visual diffs, and quality-preset evidence for every expensive system.

Work can move between these bullets as dependencies require, but no new UE-specific work should be chosen as the next slice while these gaps remain.

### Frozen Track: UE Project Asset Bridge And Visual Baseline

This track is frozen until the renderer-first acceptance gate above is met. Existing bridge code and smoke tests may remain as regression coverage, but do not spend new implementation time on parsing UE5 projects, reconstructing UE levels, exporting Unreal scenes, expanding bridge manifests, or matching UE viewport screenshots while the core renderer still lacks mainstream 3A systems.

- Keep existing bridge/project-preview code compiling and avoid breaking it during renderer work.
- Do not add new `.uasset/.umap` parsing, UE automation, project browser, scene reconstruction, export-format plumbing, or UE visual parity gates in the current renderer-first track.
- Return to this track later to enumerate `D:\UEProject`, consume bridge manifests/exported scenes, preserve stable asset ids, and compare against UE viewport references after the renderer is ready enough for that comparison to be meaningful.

### Phase 0: Render-Graph Carrying Capacity

- Add an explicit frame graph/pass description layer with names, queues, resources, active/roadmap status, and debug visibility.
- Keep the existing Forward 3D output unchanged while describing current Shadow/Main/Overlay/ImGui/Present passes through the new layer.
- Extend stats and ImGui so every future feature lands as an observable pass before becoming a complex Vulkan implementation.
- Keep legacy render pass objects alive for now; introduce dynamic rendering/transient attachments only after the graph vocabulary is in place.

### Phase 1: HDR Deferred Foundation

- Introduce render targets for HDR scene color, depth, velocity, and a compact GBuffer.
- Treat imported standalone models, benchmark material grids, and renderer stress scenes as the reference workload for deferred parity while UE scene parsing is paused.
- Start with a resource-plan contract inside the frame graph: legacy physical resources and future HDR/GBuffer/history/cache resources must be visible in ImGui and benchmark CSV before Vulkan attachment migration starts.
- Allocate a real per-swapchain-image HDR scene color target first, then migrate the rendering path to write/resolve through it in a later slice.
- Add an HDR offscreen render pass and framebuffer, then record a clear-only HDR pass before scene rendering migrates, so pipeline/framebuffer lifetime can be validated separately from visual output changes.
- Allocate the first deferred target set before the GBuffer pass exists: scene depth, velocity, albedo, normal/roughness, and material buffers.
- Add a clear-only GBuffer render pass and framebuffer over those targets before migrating opaque geometry.
- Add a first GBuffer 3D geometry pipeline that writes opaque albedo, normal/roughness, material parameters, velocity placeholder, and depth while legacy forward still owns visible output.
- Add a first fullscreen deferred-lighting pass that samples the GBuffer and writes HDRSceneColor; keep swapchain presentation on legacy forward until debug/composite parity is proven.
- Add an HDR composite/debug path that samples HDRSceneColor, tone maps it, and can replace the legacy forward scene pass through a debug view or benchmark environment switch.
- Add GBuffer debug visualizers for albedo, normal, roughness, metallic, depth, velocity, and deferred shadow visibility so deferred correctness can be inspected before it becomes the default path.
- Add the first deferred shadow parity slice: reconstruct world position from depth using inverse frame matrices, sample the existing directional shadow map from deferred lighting, and keep the legacy single shadow map as the fallback tier until the production shadow-atlas work starts.
- Split opaque 3D rendering into GBuffer generation plus deferred lighting, while preserving Forward 3D as a fallback/debug path.
- Move material data from push-constant-heavy layout toward GPU material buffers with stable material ids.
- Add debug views for GBuffer channels, velocity, depth, light complexity, and overdraw.

### Phase 2: Modern Lighting Core

- Add scene light components for directional, point, spot, rect, and emissive-light hints.
- Map imported or sidecar light/sky/atmosphere parameters into scene-owned light components with explicit unsupported-field diagnostics; UE-specific light mapping is paused.
- Add the first scene-owned primary directional light to `Scene3D`, with renderer collection into `FrameLightSet` and material-push-constant fallback kept only for compatibility paths.
- Add the first scene-owned point light data model to `Scene3D`, with light revision tracking and renderer collection into `FrameLightSet`.
- Add the first scene-owned spot light data model to `Scene3D`, encoded through local-light slots with direction/type and cone-angle payloads.
- Add the first scene-owned rect light data model to `Scene3D`, encoded through local-light slots with direction/type and width/height payloads as a first-cut area-light approximation.
- Move the first directional-light data out of material push constants and into renderer-owned frame light constants so forward, deferred, shadow projection, and debug views read one authoritative light direction/intensity source.
- Introduce an explicit renderer light scaffold (`RendererLightKind`, `RendererDirectionalLight`, and `FrameLightSet`) before GPU light buffers. This gives Phase 2 a real ownership boundary and exposes total/directional/local light counts in diagnostics.
- Upload the `FrameLightSet` into a renderer-owned GPU-visible storage buffer bound from the frame descriptor set. The first buffer tier carries one directional light, ambient/specular controls, frame light counts, and fixed local-light records used by deferred lighting.
- Add a first CPU-populated tiled light assignment scaffold over the local-light record array, then replace the population step with compute culling and feed both deferred and Forward+ passes.
- Replace the single hard-coded directional light material path with renderer-owned light buffers.
- Add environment lighting: sky irradiance, reflection probes, prefiltered cubemaps, BRDF LUT, and local reflection fallback.

### Phase 3: Production Shadowing

- Upgrade the single directional shadow map into cascaded shadow maps with stable splits, per-cascade texel snapping, bias controls, and debug views.
- Use controlled sun/sky, dense local-light, and imported standalone scenes as the quality reference for shadow softness, contact grounding, cascade stability, and atlas budgets.
- Add spot and point shadow atlas support with per-light resolution budgets, cube/dual-paraboloid policy decided by quality tier, and atlas occupancy diagnostics.
- Add filtering tiers: baseline PCF, higher-quality PCSS-style soft shadows, EVSM/variance-shadow-map experiments where format support is acceptable, and contact shadows for near-field grounding.
- Preserve current single shadow map as the Low/Fallback tier until cascades and local-light shadow atlases are benchmark-proven.

### Phase 4: Screen-Space Reflections/AO + Probe Lighting

- Add SSAO first, then deepen it toward GTAO with normal/depth-aware filtering, temporal stability, and debug views.
- Add SSR for glossy reflections, with roughness-aware tracing, thickness controls, hierarchical depth traversal, temporal accumulation, and reflection-probe fallback.
- Add sky irradiance, BRDF LUT, prefiltered environment maps, local reflection probes, and probe blending volumes.
- Add generic reflection-capture import/capture, skylight cubemaps, baked probe metadata, and post-process reflection settings through engine-owned scene data; UE-specific bridge fields are deferred.
- Add a probe GI/light-probe tier for static or slowly changing indirect diffuse lighting before considering fully dynamic GI research.

### Phase 5: Mainstream Geometry/Visibility Scalability

- Convert imported meshes into an engine-native cache with stable mesh/material ids, LOD groups, bounds, material ranges, and streaming-friendly chunks.
- Preserve source asset ids when available, LOD screen-size thresholds, lightmap/UV-channel metadata, collision-independent render bounds, and instance grouping through generic import metadata.
- Add classic LOD selection, hysteresis/fading policy, instancing, GPU-driven indirect draw preparation where it reduces CPU cost, and hierarchical occlusion using depth/Hi-Z.
- Add meshlet metadata only as a compatibility-ready asset format, not as a Nanite-style virtualized renderer.
- Keep classic mesh rendering for skeletal meshes, translucent meshes, debug geometry, and unsupported materials.

### Phase 6: Temporal AA/Upscaling And Post

- Add velocity-correct TAA, then dynamic-resolution temporal upscaling.
- Add a TAAU/FSR-style plugin boundary with history, reactive masks where useful, exposure, jitter control, sharpening, and quality presets.
- Move tone mapping, exposure, bloom, color grading, motion blur, DOF, and sharpening into graph passes.
- Match captured renderer reference settings for exposure, tone mapping, color grading, fog, bloom, DOF, and motion blur closely enough that screenshot deltas are attributable to known unsupported features rather than arbitrary SelfEngine defaults.

### Phase 7: Production Renderer Tooling And Benchmarks

- Add RenderDoc-friendly debug names and capture markers for every pass/resource.
- Add a production renderer viewport/debug mode: scene list, benchmark selection, source/imported asset inspector where applicable, missing-feature report, camera bookmarks, reference screenshot selection, and SelfEngine screenshot capture.
- Add benchmark scenes: city-scale static mesh, many lights, shadow-atlas stress, SSR/AO/probe-lighting stress, volumetric/post stress, black-hole full-screen stress, heavy post/upscale, transparency/OIT stress, and imported standalone model scenes.
- Add CSV columns per pass, per resource pool, GPU memory, shadow atlas occupancy, LOD/instance counts, light count, probe count, history resets, dynamic-resolution scale, source asset count where available, missing asset count, fallback material count, unsupported feature count, and visual-diff summary metrics.
- Add quality presets: Cinematic, Epic, High, Medium, Low, and FPS-first. Every expensive system must have a measurable fallback.

### Phase 8: Renderer Feature-Complete Stabilization

- Promote deferred HDR to the default visible path only after the fallback/debug forward path, imported generic assets, material parity, lighting, shadows, probes, translucency, post, temporal stability, and diagnostics all pass smoke coverage.
- Run feature-combination benchmarks instead of single-feature demos: many lights plus CSM plus SSAO/SSR/probes, volumetrics plus transparency plus bloom, TAA/upscaling plus motion vectors, and imported-model material grids under multiple quality presets.
- Build renderer-owned reference captures and visual diffs for built-in stress scenes and generic imported scenes. These captures become the readiness proof for returning to UE integration later.
- Produce a single missing-feature report that lists unsupported renderer features, degraded fallbacks, disabled quality tiers, and known visual mismatches before any UE project-preview work resumes.

### Advanced Research Backlog

These systems are intentionally postponed until the mainstream AAA track above is usable, benchmarked, and debuggable:

- Nanite-class virtualized geometry: cluster hierarchy, cluster streaming, software/compute visibility, visibility buffers, material resolve, and automatic LOD at film-scale triangle counts.
- Lumen-class dynamic GI/reflections: surface cache/card capture, signed-distance-field or BVH tracing, software/hardware ray tracing tiers, temporal denoise, and fully dynamic multi-bounce lighting.
- Full Virtual Shadow Maps: virtual page tables, physical page atlases, page marking, invalidation/caching, directional clipmaps, local-light pages, and Nanite-scale scene integration.
- Hardware ray tracing path: BLAS/TLAS build policy, shader binding table management, ray-traced shadows/reflections/GI, denoisers, and raster fallbacks.
- Native TSR-class upscaler: deep history rejection, reactive/transparency masks, exposure-coupled resolve, async-compute scheduling, and dynamic-resolution integration beyond the first TAAU/FSR-style path.

## Implementation Rules

- Keep current demos running at every phase: `SelfEngine`, `SelfEngineForward3D`, and `SelfEngineBlackHole`.
- Keep existing UE bridge/project-preview code compiling when practical, but do not use it as a current milestone gate or primary validation scene.
- UE-specific changes are allowed only when renderer work breaks existing bridge compilation/regression coverage. Do not start new bridge functionality while mainstream renderer features are incomplete.
- Treat renderer reference scenes, imported standalone models, and stress benchmarks as the current product reference. New lighting, material, shadow, post, translucency, visibility, and temporal features must include screenshots/CSV/log diagnostics tied to renderer-controlled scenes and feature toggles.
- Do not close a phase with only a pretty screenshot. A phase closes when the feature has a visible path, an off/fallback path, debug view or ImGui control, CSV/frame-graph evidence, and at least one benchmark/smoke that exercises it.
- Do not remove the existing Forward 3D path until the deferred path can render imported textured models, shadows, debug views, and benchmark scenes.
- Use handcrafted benchmark scenes as valid current renderer proof when they are built to stress a specific mainstream 3A feature and have measurable diagnostics.
- Preserve source/import identity when available through import, renderer material records, diagnostics, screenshots, and CSV/log output so a visual mismatch can be traced back to the originating asset/material/light/scene record.
- Any unsupported material feature, light type, post-process field, texture format, geometry feature, or scene component in the active renderer/import path must produce an explicit fallback record instead of silently becoming a generic material or disappearing.
- Prefer adding thin renderer-owned concepts first: pass descriptions, render targets, material ids, light ids, history ids. Add heavyweight GPU algorithms only after the interfaces exist.
- Every phase must add a visible debug mode or benchmark metric; no invisible renderer rewrite counts as complete.
- Use Vulkan 1.3 as the baseline and gate optional mainstream paths behind feature checks: descriptor indexing, dynamic rendering, synchronization2, subgroup operations, and async compute. Mesh shaders and ray tracing remain advanced-research feature gates, not current-track requirements.
- Third-party libraries are acceptable by default when they are mature, reasonably licensed, actively maintained, and hidden behind narrow SelfEngine-owned interfaces. Add them through CMake/FetchContent or local third-party folders in a way that keeps builds reproducible, and record the reason for adoption in this plan or a small ADR when the dependency affects a major subsystem.
